## Build flow

```
build:docs:  pnpm --filter starlight-docs build  ‚Üí  starlight/dist/
             cp -R starlight/dist  ‚Üí  public/docs/
build:app:   astro build (root)  ‚Üí  dist/  (sweeps public/* including docs/)
```

Sequential ‚Äî main consumes `public/docs/`. Don't parallelize without restructuring (see dev-build skill).

## Key config

| Where | What | Why |
|---|---|---|
| `starlight/astro.config.mjs` | `base: '/docs'` | Prefixes Starlight-generated links/assets |
| `starlight/astro.config.mjs` | `server.port: 4322` | Avoid main app collision |
| `starlight/astro.config.mjs` | **No `outDir` override** | Outside-project outDir breaks Astro image cache (intermediate written to `outDir/_astro/`, read from `<project>/.astro/_astro/`) |
| `pnpm-workspace.yaml` | `packages: [., starlight]` | Single `pnpm install` from root |
| `pnpm-workspace.yaml` | `allowBuilds: { sharp, esbuild }` | Required for postinstall |
| `.gitignore` | `/public/docs/` | Build artifact |

## Link prefixing

Starlight does **not** auto-prefix absolute-path markdown links with `base`. We fix this via a `remarkBaseLinks()` remark plugin defined inline in `starlight/astro.config.mjs` and wired into `markdown.remarkPlugins`. It walks every body markdown link node ‚Äî if `url` starts with `/` and doesn't already start with `/docs/`, it prepends `/docs`. Covers regular `.md`/`.mdx` doc pages; idempotent.

What it does NOT cover:
- **Frontmatter** is not part of the markdown AST. Splash `hero.actions[].link:` and any custom URL frontmatter field must be written as full `/docs/...` paths.
- **Splash template body markdown** (`template: splash` pages) ‚Äî Starlight renders these through a different pipeline that bypasses `markdown.remarkPlugins`. There is no splash page today; if one is added, its body links must be written as full `/docs/...` paths. Tested empirically: regular doc pages get auto-prefixed, splash pages do not.

If you add a new splash page, write all of its body links + hero actions with explicit `/docs/` prefix. Validate via `curl -s <url> | grep -oE 'href="[^"]+"' | sort -u`.

If you add a new frontmatter field that contains a URL path, write it base-prefixed (the plugin won't help).

## Other gotchas

- Astro `redirects` config: *source* gets `base` applied (so `/` resolves to `/docs/`), but *destination* does **not** ‚Äî write the full path (`/docs/foo/`, not `/foo/`). Current: `/` ‚Üí `/docs/start-here/welcome/`. **File-based routes take precedence over `redirects:`** ‚Äî keeping `index.md` would silence the redirect, so the splash file was removed.
- `pnpm install` from inside `starlight/` walks up to workspace root anyway ‚Äî always install from repo root.
- `build:docs` uses `rm -rf` + `cp -R` ‚Äî Unix-only.
- No link from `/` ‚Üí `/docs/` by design.
- Content schema is extended (`starlight/src/content.config.ts`) for custom keys: `stability` (`stable` | `working` | `experimental`), `last_synced_with` (string), `sources` (array of strings). Required by the docs content. Removing these is a breaking change.

## Adding content

- Docs: `.md`/`.mdx` in `starlight/src/content/docs/`. Path = URL (relative to `/docs/`). Sidebar in `starlight/astro.config.mjs` or `autogenerate`.
- Static API/spec assets: `starlight/public/specs/`. Path = URL (relative to `/docs/specs/`).
- Docs images: `starlight/src/assets/`, ref relatively in MDX.
- Main app pages: `src/pages/`.
- Main app static: `public/` ‚Äî **never `public/docs/`**.
- Project config examples: `config/`. Commit examples only; keep environment-specific secrets out of the repo.
