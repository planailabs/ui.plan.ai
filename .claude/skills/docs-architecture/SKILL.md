# Docs architecture

This repo runs **two Astro projects in one pnpm workspace**:

| Project | Path | Serves | Dev port |
|---|---|---|---|
| Main app | `/` (root) | `ui.plan.ai/` | `:4321` |
| Starlight docs | `/starlight/` | `ui.plan.ai/docs/` | `:4322` |

They share a domain but not a build. Each has its own `package.json` and `astro.config.mjs`. The workspace is just plumbing — there is no shared code or shared layout between them by design (chosen: independent branding).

## File layout

```
/
├── package.json              # root: main Astro app + orchestration scripts
├── astro.config.mjs          # main app config (currently empty)
├── pnpm-workspace.yaml       # declares both root and starlight/ as packages
├── src/pages/                # main app pages (one index.astro for now)
├── public/                   # main app static assets
│   └── docs/                 # ⚠ generated by build:docs — gitignored
├── dist/                     # final merged build output (gitignored)
│   └── docs/                 # docs land here after main build runs
└── starlight/
    ├── package.json          # name: "starlight-docs"
    ├── astro.config.mjs      # base: '/docs', server.port: 4322
    ├── src/content/docs/     # all docs content lives here (.md, .mdx)
    ├── src/content.config.ts # starlight content collection schema
    ├── src/assets/           # docs images (referenced from .mdx)
    └── dist/                 # docs build output (gitignored)
```

## Build pipeline

```
pnpm build
  └─► pnpm build:docs
        ├─► pnpm --filter starlight-docs build  ──►  starlight/dist/
        └─► cp -R starlight/dist  ──►  public/docs/
  └─► pnpm build:app
        └─► astro build (root)                  ──►  dist/
                                                       ├── index.html  (main)
                                                       └── docs/...    (from public/docs)
```

Order matters: docs must build before main, because `astro build` at root sweeps `public/` into `dist/`.

## Dev workflow

`pnpm dev` runs both servers via `concurrently`:

- Main app at http://localhost:4321/ (live HMR)
- Docs at http://localhost:4322/docs/ (live HMR)

Docs HMR comes from the **Starlight dev server, not the main app**. The `public/docs/` static copy is whatever was last built — main app at `:4321/docs/*` will be stale or 404. Always go to `:4322/docs/` while iterating on docs content.

## Scripts (root `package.json`)

| Script | What it does |
|---|---|
| `dev` | Both projects in parallel (concurrently) |
| `dev:app` | Only main Astro (`:4321`) |
| `dev:docs` | Only Starlight (`:4322`) |
| `build` | Docs → main, ordered |
| `build:app` | Only main app (`astro build`) |
| `build:docs` | Build Starlight + copy `starlight/dist` → `public/docs` |
| `preview` | Preview the merged `dist/` |

## Key configuration

**`starlight/astro.config.mjs`:**
- `base: '/docs'` — Starlight prefixes all *internal* links/assets with `/docs/`. Required so links work when served at the subpath.
- `server.port: 4322` — avoid collision with main app at `:4321`.
- `outDir` is **not** set. Initial attempt with `outDir: '../public/docs'` hit a known Astro image-pipeline bug (cached webp written to `outDir/_astro/` but read from `<project>/.astro/_astro/`). Always build to the default `dist/` and copy.

**Root `.gitignore`:**
- `/public/docs/` is gitignored — it's a build artifact. Never commit it.
- `dist/`, `.astro/`, `node_modules/` ignored everywhere.

**`pnpm-workspace.yaml`:**
- `packages: [., starlight]` — makes `starlight/` a workspace member. Needed so a single `pnpm install` at root sets up both, and so `pnpm --filter starlight-docs <cmd>` works.
- `allowBuilds: { esbuild, sharp }` — sharp is Starlight's image-optimization dep.

## Gotchas

1. **User-written links don't get the `base` prefix.** Astro only prefixes assets and routes *it* generates. A `link: /guides/example/` in MDX frontmatter or `<a href="/guides/example/">` stays exactly as written — it'll 404 in production. Either write internal links as slugs Starlight resolves (`slug: guides/example`, Starlight's `<LinkCard href="/guides/example/">` with base awareness), or include `/docs/` explicitly.

2. **Don't run `pnpm install` from inside `starlight/`** expecting an isolated install. pnpm walks up to the workspace root. Run `pnpm install` from repo root; deps for both projects resolve together.

3. **The `pnpm --filter` target is the `name:` field**, not the directory. `starlight/package.json` has `"name": "starlight-docs"` — that's the filter handle. Renaming the field will break root scripts.

4. **`build:docs` uses `rm -rf` and `cp -R`** — not Windows-portable. If a Windows contributor ever joins, swap to `cpy-cli` or similar.

5. **Adding a redirect from `/` → `/docs`** would require a route in main `src/pages/`. User explicitly said they don't want a link from main to docs; the `/docs` URL is meant to be discovered/shared directly.

## Adding docs content

- New page: drop a `.md` or `.mdx` in `starlight/src/content/docs/`. Directory structure → URL path (relative to `/docs/`).
- New sidebar entry: edit the `sidebar` array in `starlight/astro.config.mjs`, or rely on `autogenerate: { directory: '...' }` for a folder.
- New image: put in `starlight/src/assets/` and reference relatively from the `.mdx`. Sharp will optimize during build.

## Adding main app content

- New page: drop an `.astro`/`.md`/`.mdx` in `src/pages/`.
- Static asset: drop in `public/` — **but never in `public/docs/`**, that path is owned by the docs build and gets `rm -rf`'d on every `build:docs`.
